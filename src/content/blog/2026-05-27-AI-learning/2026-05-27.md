## Week 1 & 2（5.26-5.30）

第一周把 Week 1 和 Week 2 的内容都做完了。

Week 1 目标是跑通 LLM 基础调用，Week 2 是实现最小 Agent Loop——模型多轮调用工具，直到任务完成才退出。

**1：provider 配置方式**

最开始对每个 provider 写了 if/else，后来改成字典注册表，新增只需加一行。

另外 `create_client()` 最初返回 `(client, settings)` 两个值，到处要一起传。后来发现封装成一个 `LLMClient` dataclass 更干净：

```python
@dataclass
class LLMClient:
    client: OpenAI
    model: str
```

**2：tool 的 schema 和 handler 分散**

最开始 tool 的 JSON Schema（给模型看）和实际执行函数（给代码用）是两个独立变量，容易对不上。后来用 `@dataclass` 绑在一起：

```python
@dataclass
class ToolDefinition:
    name: str
    schema: ChatCompletionToolParam  # 给模型
    handler: Callable[..., str]      # 给代码执行
```

**3：agent loop 反复调用同一个工具**

在测试的时候发现跑 `先 echo 这句话，再解释你收到的内容`，工具被调用了 14 次才停下。

原因：每轮调用完工具后，只把 tool result 加入 messages，忘了先把 **assistant 消息**（包含 tool_calls）也加进去。模型每轮看不到自己上轮的 tool call，以为任务还没开始，就一直重复调用。

正确顺序：

```plaintext
user      → 用户输入
assistant → { tool_calls: [...] }   ← 这条必须先加，否则模型失忆
tool      → 执行结果
assistant → 最终回答
```

## Week 3 & 4（5.31-06.07）

本周仍然是完成了 2 周的任务。

- Week 3 是希望通过不断修改 prompt 和 tool design 来理解其对 Agent 的影响。
- Week 4 是开始实现和 Code Agent 有关的 tool，并且开始理解 tool 的组合使用以及各种细节。

本周主要是学习到了 [ripgrep](https://github.com/BurntSushi/ripgrep) 的使用，以及 tools 涉及到的安全控制（文件路径，文件数量等）。

还踩了一个坑，就是当 AI 输出文件内容的时候可能会自己对内容进行处理。这其实是不好的行为，这会让实际的输出和你的预期有偏差，比如吞掉你打印出的行号。我们需要在 `SYSTEM_PROMPT` 里加以限制。

```python
SYSTEM_PROMPT = """
# ...
- When displaying tool output (file contents, search results), always show it verbatim. Do not add, remove, reformat, or modify any content including line numbers and comments."""
```

## Week 5（06.08-06.21）

这两周只完成了 Week 5 的任务。 Week 5 的主要任务是实现一个大而全的 tools - command runner，更多的是需要做对 tool 做安全的限制，比如目录的检查，运行命令的限制，超时控制等。

**1：command runner的命令解析**

最开始直接让 LLM 传入一整个 command，使用 `shlex.split(command)` 来解析，但是这里就已经有了很多问题，比如如何确定解析后的字符串中，哪一个是工具，哪一个是参数，哪一个是路径。特备是我们还需要对工具/目录做出限制。
AI 的方案让我豁然开朗，即这本身就是一件偏自然文本理解的事情，交给 AI 做才是最合理的。

所以我们需要在 tool 的定义中直接把 command 和 args 分开定义。

```python
# old version
@dataclass
class CommandRunner(BaseModel):
    command: str

# new version
@dataclass
class CommandRunner(BaseModel):
    command: str
    args: list[str] = Field(default_factory=list)
    working_directory: str = "."
```

当然这里 command 本身是很好拆分的，问题还是怎么获取其中的目录。现在我们仍需要从 args 中获取，但是 args 中可能会有很多参数，比如 `-f`，`--file`，`-o`，`--output` 等，我们需要把这些参数都去掉，只保留目录。

为什么不直接让 LLM 把目录也拆分出来当成一个参数呢？这里主要是如果拆分出来，我们可能会遇到多个文件/目录的情况，而且文件/目录在参数中的位置也是不确定的。所以本质上我想我们不应该使用一个大而全的 tool，而是使用多个小而精的 tool，这样就可以根据命令本身的规则拆分具体的文件/目录出来。

**2：tools 的注册**
这个在前几周就遇到了，就是每次切换我们的 tools 集合，我都要修改 `chat_basic.py` 里的 `SYSTEM_PROMPT` 和 `agent.py` 里的 `TOOLS` 变量。这明显有优化的空间。

所以比较好的方式是在 tools 目录下创建 `__init__.py`， 然后定义每周的 `TOOLS` 和 `DEFAULT_TOOLS`，然后在 `chat_basic.py` 里调用这个`DEFAULT_TOOLS`变量。

```python
# src/mini_coding_agent/tools/__init__.py
WEEK4_TOOLS = [LIST_FILES_TOOL, READ_FILE_TOOL, SEARCH_CODE_TOOL]
WEEK5_TOOLS = [COMMAND_RUNNER_TOOL]
DEFAULT_TOOLS = WEEK5_TOOLS
```

同样的，对于 `SYSTEM_PROMPT` 我们不要在里面直接指定,而是让 LLM 自己去理解 TOOL 的 schemas

```python
SYSTEM_PROMPT = """
## Tool use

Use the provided tool schemas as the source of truth for available tools and arguments.
Prefer read-only repo tools before command execution.
```
