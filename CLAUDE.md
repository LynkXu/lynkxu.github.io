# CLAUDE.md

本文件用于补充项目的实现级协作规范。

- 设计语义与品牌方向以 [.impeccable.md](/Users/link/Code/lynkxu.github.io/.impeccable.md) 为上层背景。
- 本文件聚焦“现有代码风格的基本框架”和“每次修改后的强制检查项”。
- 除非用户明确要求改风格，否则后续任务都不能脱离这里定义的框架。

## 1. 当前项目的风格基线

以下基线已结合当前代码与实际页面表现整理，主要参考：

- [src/styles/style.scss](/Users/link/Code/lynkxu.github.io/src/styles/style.scss)

- [src/components/Header.astro](/Users/link/Code/lynkxu.github.io/src/components/Header.astro)
- [src/components/Footer.astro](/Users/link/Code/lynkxu.github.io/src/components/Footer.astro)

### 1.1 整体气质

- 内容优先，首先保证阅读效率，而不是追求强装饰。
- 版面偏克制、留白稳定、对比柔和，避免突兀的大色块和过饱和强调色。
- 页面观感接近“纸面阅读”而不是“产品后台”或“营销落地页”。
- 允许专题页和二级页面相对博客正文有轻微独立风格，例如运动、摄影、书影音等页面可以更偏数据化、画廊化或专题化。
- 这种差异应建立在同一套排版、颜色、间距和交互语法上，避免变成另一套完整设计语言。

### 1.2 字体系统

- 正文、标题默认使用 `var(--serif-font)`，这是当前站点最核心的阅读气质来源。
- 导航、标签、日期、说明文字、按钮、元信息默认使用 `var(--ui-font)`。
- 代码、数字标签、技术性标记优先使用 `Lilex` 或既有 monospace 栈。
- 不要随意引入新的字体家族，也不要在普通正文区域把字体切成 UI 字体。
- 中英文混排时优先保持现有字体栈，不要为了局部效果临时替换某一段字体。

### 1.3 字号与层级

- 正文基础字号以 `--text-md` 为主，当前值约为 `0.95rem`。
- 标题层级使用现有 token：`--text-lg`、`--text-xl`、`--text-2xl`，不要绕开 token 重新造一套。
- 元信息字号通常落在 `0.68rem` 到 `0.78rem` 之间，列表标题多在 `0.9rem` 到 `1.1rem`。
- 新增文字样式时，优先复用已有层级；只有现有层级无法表达时，才允许新增局部尺寸。
- 同一区块内不要同时出现过多相近字号，否则会破坏层级辨识。

### 1.4 行高与字距

- 正文行高以 `--lh-body` 为主，当前主值约为 `1.8`，长文可读性优先。
- 标题行高使用更紧的 `--lh-tight` 系列，保持标题凝聚力。
- 标题字距基本保持自然或轻微负字距，正文不要做明显压缩字距。
- 元信息可以有非常轻的字距扩展，但不能做强烈全大写式处理。
- 一切行高和字距调整都应服务于阅读节奏，不能只为了“看起来更设计”。

### 1.5 容器与宽度

- 阅读主容器遵循现有宽度 token：`--width`、`--content-width`、`--blog-shell-width`、`--blog-content-width`、`--blog-readable-width`。
- 普通阅读内容保持窄列，不要轻易把文章主体改成大宽屏铺满。
- 首页、归档、标签等列表页延续当前“内容列 + 留白”的阅读式骨架。
- 文章正文的最佳可读宽度继续以 `--measure-readable` 和既有 blog 容器为准。
- 只有图片、图库、数据可视化等明显需要横向空间的模块，才允许使用更宽容器。

### 1.6 间距系统

- 间距必须优先使用现有 spacing token：`--space-2xs` 到 `--space-4xl`。
- 页面外边距优先使用 `--page-inline` 与容器 token，不要大量写死 `margin-left/right`。
- 大区块之间的节奏依赖 `clamp()` 和 token 的组合，这种写法应继续保持。
- 新增组件时先对齐相邻模块的 vertical rhythm，再决定自己的 `gap`、`padding`、`margin`。
- 避免无依据的随机值，如 `17px`、`29px` 这类脱离系统的局部修补。

### 1.7 卡片、边框与阴影

- 当前项目卡片偏轻纸感：细边框、极轻阴影、低圆角、低存在感。
- 默认沿用 `--paper-card-bg`、`--paper-card-border`、`--paper-card-shadow`、`--paper-card-radius`。
- 不要把卡片做成强产品化的高圆角浮层，也不要使用厚重投影。
- 当前首页和列表页的卡片 hover 只做轻微位移，保持克制。
- 装饰线、边角标记、ledger 感是现有视觉语言的一部分，新增模块应优先兼容。

### 1.8 颜色系统

- 颜色必须优先使用现有语义 token，如 `--text-color`、`--heading-color`、`--gray-color`、`--accent-color`、`--link-color`。
- 当前配色基于低饱和、轻纸面、轻墨色，不要突然引入鲜艳品牌色主导页面。
- 链接、标签、说明文字的颜色对比要服从当前柔和体系，而不是走强按钮感。
- 亮色和暗色主题都已存在，新增样式要考虑两套主题下是否都成立。
- 除非是状态语义，否则不要滥用 `warning/danger/success` 色。

### 1.9 动效与交互

- 动效只用于轻反馈，不承担主要信息层级。
- hover 位移和淡入淡出保持当前幅度，通常在 `1px` 到 `2px` 的移动范围内。
- 动效时长继续维持在约 `0.2s` 到 `0.35s` 的克制区间。
- 所有新增动效都必须兼容 `prefers-reduced-motion`。
- 不要引入炫技型大范围弹跳、旋转、视差或持续动画。

## 2. 后续修改的执行准则

### 2.1 修改前

- 先看目标页面、相邻页面、对应布局和全局样式，确认已有模式再动手。
- 优先复用现有 token、现有容器、现有卡片语法、现有标题层级。
- 如果一个改动会影响多个页面，应先确认它属于全局规律还是页面例外。

### 2.2 修改时

- 能放进全局 token 的，不要写成单页 magic number。
- 能复用已有类或已有视觉语法的，不要重新发明近似样式。
- 新增局部样式必须解释它为什么不是复用现有系统。
- 即使是小改动，也不能破坏字体系统、宽度系统、间距系统的稳定性。

### 2.3 修改后的底线

- 后续任务的修改都要在当前风格框架内完成，不能把页面改成另一套设计语言。
- 若确实需要越过这套框架，必须把它视为“风格变更”而不是“普通修改”。

## 3. 每次修改后的强制视觉检查

凡是涉及页面、组件、排版、颜色、间距、交互的改动，完成后都必须检查实际效果，不能只看代码。

### 3.1 必做检查动作

- 启动本地预览或开发服务器，打开实际页面检查。
- 至少检查受影响页面的桌面视图。
- 只要改动涉及布局、字号、换行、导航、卡片、列表、图片、弹层、按钮，就必须额外检查移动端视图。
- 如果改动影响全局样式或布局基类，要抽查至少一个首页/列表页和一个正文页。

### 3.2 必查项目

- 字体一致性：正文、标题、元信息、代码字体是否仍然各归其位。
- 字号一致性：同类信息是否仍属于同一层级，没有忽大忽小。
- 行高一致性：正文是否舒展可读，标题是否过松或过挤。
- 宽度合理性：正文行长是否失控，容器是否突然过宽或过窄。
- 间距合理性：模块之间是否保持统一节奏，是否出现某一块异常拥挤或松散。
- 对齐一致性：标题、日期、正文、标签、页脚、导航是否仍沿同一视觉边线。
- 卡片一致性：边框、阴影、圆角、hover 反馈是否仍属于当前纸感体系。
- 颜色一致性：文字、链接、分隔线、背景是否仍是低饱和克制风格。
- 换行质量：中英文混排是否出现难看的断行、拥挤、悬空标题或过长孤行。
- 主题兼容性：若改动触及颜色、背景、边框，需确认浅色/深色主题都正常。

### 3.3 未通过时的处理

- 如果视觉结果与现有风格不一致，继续调整，不以“功能已实现”为完成标准。
- 如果代码正确但观感别扭，优先修观感。
- 如果某项调整会破坏另一项一致性，优先回到 token 和现有模式重新收敛。

## 4. 默认工作方式

- 默认把“代码正确 + 风格一致 + 实际效果已检查”视为完成。
- 默认在回复中说明本次是否做了实际页面检查，以及检查了哪些视图。
- 默认不提交只在代码层面成立、但未验证真实页面效果的样式修改。
